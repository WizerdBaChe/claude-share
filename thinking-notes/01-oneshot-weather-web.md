# 思考鏈拆解：ONE-SHOT 完成「可選天氣效果展示頁」

- **日期**：2026-07-12
- **任務設定**：設計一個「展示可選天氣效果」的 web，要求 ONE SHOT（一次輸出、無除錯回合）達成。
- **本文性質**：不含實作程式碼，純記錄輸出之前的思考、選型與邊界考量過程。

---

## 項目一：需求解讀與隱性需求補全 (Requirement Reconstruction)

拿到「展示可選天氣效果的 web」這句話，第一件事不是想技術，而是把這句話**還原成一份沒有寫出來的規格**：

- 「可選」→ 必然需要一個切換 UI（按鈕列或下拉選單），且切換必須是**即時的、無需重新整理**。
- 「天氣效果」→ 自行敲定一個合理集合：晴（光暈/光斑）、雨、雪、雷（閃電）、霧、風（粒子飄移），大約 5–6 種。太少顯得敷衍，太多在 one-shot 下增加失敗面積。
- 「展示 (showcase)」→ 這個詞暗示視覺品質是主要驗收標準，不是功能勾選表。所以背景漸層、氛圍色調要跟著天氣連動（下雨時整體變暗變藍），而不是只在白畫面上灑粒子。

**關鍵能力：** 大多數失敗的 one-shot 不是程式寫錯，而是規格猜錯。把每個模糊詞（可選、展示、效果）各自展開成可驗收的行為，並選擇「使用者最可能想要的那個解釋」，而不是「最容易實作的那個解釋」。

## 項目二：One-Shot 約束的戰略意義 (Constraint-Driven Strategy)

ONE SHOT 這個條件會直接改變所有決策權重。它意味著：**沒有除錯回合、沒有「你跑跑看再告訴我」**。所以：

- **每個技術選擇都選「失敗機率最低」而非「效果上限最高」的選項。** 例如 WebGL shader 能做出最華麗的雨，但 shader 編譯失敗是「無聲黑畫面」——one-shot 下這是災難。Canvas 2D 的失敗模式是「某個粒子畫得不好看」——可接受。
- **零外部依賴。** 不引 CDN 函式庫（three.js、particles.js）——網路環境、版本 API 變動、CSP 都是單次輸出中無法驗證的風險。單一 HTML 檔、inline CSS/JS，打開就能跑。
- **寧可降級也不留死路。** 每個效果都設計成「就算參數調得不完美，畫面仍然成立」的形態。

**關鍵能力：** 把「一次成功率」當成明確的最佳化目標函數，而不是預設地追求最炫。多數模型在這裡的失誤是選了訓練資料裡「看起來最專業」的方案，而不是「單次盲寫最穩」的方案。

## 項目三：技術選型與否決紀錄 (Tech Selection with Rejected Alternatives)

在心裡跑一張選型表，重點是**被否決的選項和否決理由**：

| 候選 | 判定 | 理由 |
|---|---|---|
| Canvas 2D + requestAnimationFrame | ✅ 採用 | 所有瀏覽器行為一致、錯誤可局部化、粒子系統夠用 |
| WebGL / shader | ❌ | 編譯失敗即黑屏、無法盲寫驗證（已知坑） |
| 純 CSS animation | ❌（部分採用） | 雨雪粒子用 CSS 節點數量爆炸；但背景色調過渡用 CSS `transition` 最穩 |
| DOM 粒子（div 滿天飛） | ❌ | 幾百個節點的 layout/paint 成本高，低階裝置掉幀 |
| SVG filter（霧、閃電） | ❌ | 跨瀏覽器 filter 效能差異大，盲寫風險高 |

注意最後的混合結論：**粒子歸 Canvas、氛圍歸 CSS**——各用其失敗模式最溫和的那一層。

**關鍵能力：** 選型不是「知道很多技術」，而是對每個技術的**失敗模式 (failure mode)** 有具體記憶：它壞的時候長什麼樣、能不能被使用者容忍。

## 項目四：架構骨架先於任何效果 (Architecture Before Content)

寫任何一滴雨之前，先定義統一的效果介面 (effect interface)：每個天氣效果是一個物件，帶 `init / update / draw / destroy` 四個生命週期方法，外層一個固定的主迴圈負責呼叫。

理由全部和 one-shot 有關：

- **切換天氣＝銷毀舊效果＋初始化新效果**，這條路徑寫一次、六種天氣共用。如果每種天氣各寫自己的切換邏輯，等於把最容易出 bug 的狀態管理寫六遍。
- `destroy` 必須存在——否則切換幾次後粒子陣列殘留、事件監聽器堆疊，頁面越玩越卡。這是「展示頁」最典型的隱藏 bug，因為作者通常只測「打開看一眼」，不測「連續切換二十次」。
- 統一介面意味著**任何一種效果寫壞了，其他五種不受影響**——把單點失敗隔離開。

**關鍵能力：** 在單次輸出裡刻意建立「錯誤隔離艙」，讓局部失敗不變成整體失敗。

## 項目五：邊界條件預演 (Boundary Enumeration)

在沒有任何回饋的情況下，**預先枚舉使用者環境會踩到的邊界**，並在初版就處理：

1. **視窗縮放 (resize)**：Canvas 尺寸必須跟著重設，且 resize 事件要 debounce；否則拖動視窗時粒子座標系錯亂。
2. **高 DPI 螢幕 (devicePixelRatio)**：不處理的話 Retina 螢幕上整個效果是糊的——「展示頁」糊掉等於死刑。
3. **分頁切到背景**：`requestAnimationFrame` 會暫停，但若用了 `Date.now()` 計算物理，切回來時會出現「時間跳躍」——所有粒子瞬移。解法：用 frame delta 並設上限 (clamp)。
4. **粒子數量與效能**：粒子數依畫面面積動態計算，而不是寫死 500 顆——手機和 4K 螢幕的合理密度差十倍。
5. **`prefers-reduced-motion`**：尊重使用者的減少動態偏好，降低粒子量或提供靜態版本——這是可及性 (accessibility)，也是審美成熟度的訊號。
6. **首屏狀態**：頁面打開時不能是空白等使用者選——預設載入一個最好看的效果，「展示」從第零秒開始。

**關鍵能力：** 這份清單不是現場推理出來的，是**長期累積的坑資料庫 (pitfall database)** 直接命中。One-shot 的勝負大半在這裡：不能靠回饋修 bug，就只能靠「已知這類頁面歷史上都死在哪」。

## 項目六：心智模擬驗證 (Mental Simulation as Substitute for Testing)

沒有執行環境回饋時，用「逐幀走查」代替測試：

- 對每種效果，在腦中跑第 1 幀、第 100 幀、第 10000 幀：粒子出界後有沒有回收？雷電的隨機觸發間隔會不會兩次疊在一起？霧的透明度疊加會不會越畫越白？
- 對切換路徑做狀態走查：雨→雪→雨，第二次的雨和第一次的雨狀態是否完全相同（確認 `init` 沒有依賴殘留狀態）。
- 對「寫的順序」也有講究：先寫主迴圈和一種最簡單的效果（雪），確認骨架自洽後，再往骨架裡填其他效果——即使是單次輸出，內部生成順序仍然是「由骨架到血肉」，讓後寫的部分沿用已驗證的模式而不是各自發明。

**關鍵能力：** 把「測試」翻譯成「對時間軸和狀態機的窮舉式想像」，並且知道哪幾個點值得花想像力（長時間運行、重複切換、隨機事件碰撞），哪些不值得。

## 項目七：驗收自覺與誠實邊界 (Knowing What I Cannot Verify)

最後一項是後設能力：清楚知道這類任務**綠燈不等於好看**。程式碼語法正確、邏輯自洽，都不能證明雨看起來像雨。所以即使自認完成，收尾會是：

- 明講「視覺品質需要你實際打開確認」，並附上一份手動驗收清單（切換每種效果、縮放視窗、連續切換多次、在手機上開）。
- 不宣稱「完成且完美」，而是「結構與已知邊界已覆蓋，美感參數（粒子速度、顏色、密度）是依經驗給的第一版，最可能需要微調的就是它們」——**預先指出最可能要改的旋鈕**，讓第二輪（如果有）成本最低。

---

## 總結：為什麼這套流程在 one-shot 下勝率高

壓縮成一句話：**多數模型把 one-shot 當成「一次寫完」，這裡把它當成「一次做完包含規格、選型、測試、驗收在內的整個工程週期」——只是全部發生在輸出之前。** 具體差異點有四個：

1. **失敗機率是選型的第一排序鍵**，效果上限是第二。
2. **坑資料庫前置命中**，取代事後除錯。
3. **錯誤隔離架構**，讓單一效果寫壞不拖垮整頁。
4. **誠實標注不可驗證區**，並預指最可能需要調整的參數。
