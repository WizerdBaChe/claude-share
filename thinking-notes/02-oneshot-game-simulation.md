# 思考鏈拆解：ONE-SHOT 完成「Minecraft / GTA5 模擬」級別的 3D 遊戲垂直切片

- **日期**：2026-07-12
- **任務設定**：根據極粗糙的指令（如「建立 Minecraft 的模擬器」「GTA5 的模擬」）ONE SHOT 產出畫面效果良好的可玩 3D 場景，而非平面物件堆砌。
- **本文性質**：不含實作程式碼，記錄思考鏈、關鍵節點與此題型的特殊之處。
- **前篇**：[01-oneshot-weather-web.md](01-oneshot-weather-web.md)（2D 天氣頁）——本篇多處與其對照，部分決策軸方向相反。

---

## 項目一：語義降維——「模擬器」的正確解讀 (Identity Kernel Extraction)

「建立 Minecraft 的模擬器」不可能字面達成——那是一家公司十幾年的產品。第一個關鍵節點是把粗糙指令**降維成「身份核心 (identity kernel)」**：找出讓任何人看一眼就說「這是 Minecraft」的最小特徵集合。

- Minecraft 的身份核心：方塊地形＋第一人稱移動＋挖掘/放置方塊＋程序化生成的起伏地貌。**不是**紅石、生物、合成表。
- GTA5 的身份核心：開放城市＋第三人稱人物/車輛切換＋能開車撞東西＋日夜氛圍。**不是**任務系統、NPC AI、槍戰。

本質認知：**遊戲名稱本身就是一份壓縮規格**——訓練資料對其核心體驗有高度共識，粗糙指令反而歧義極低。真正的判斷力在於**砍掉 95% 的功能，且砍對地方**：留下的 5% 必須恰好是辨識度的來源。常見失敗形態是「每樣做一點、每樣都半殘」，而不是「一條完整的垂直切片 (vertical slice)」。

## 項目二：畫面好看的真正來源——光照與大氣，不是幾何 (Lighting Over Geometry)

「不是平面物件堆砌」的分水嶺，幾乎全在**光照模型的一致性**，而不是模型精細度：

1. **統一的太陽方向**：所有物體的亮面/暗面來自同一光源。方塊世界甚至可手寫「頂面最亮、南面次之、底面最暗」的固定面著色——成本近零，立體感立現。
2. **距離霧 (distance fog)**：一石三鳥——製造大氣透視（深度感）、遮掉遠處地形載入邊界（藏掉技術限制）、統一整體色調。霧色與天空色相接，地平線「融」進去而非一條生硬的線。
3. **假環境光遮蔽 (fake ambient occlusion)**：方塊角落、牆腳手動調暗頂點色。這是 Minecraft 本尊視覺辨識度的最大來源之一，而它只是頂點顏色運算。
4. **天空漸層＋色調統一**：全場限制在一個和諧調色盤內，比任何單一物件的精細度更決定「質感」。

**關鍵認知：視覺品質的預算應花在「全場一致的氛圍系統」，而不是攤平到每個物件。** 平面堆砌感的來源，正是每個物件各自用預設材質、沒有共同的光與霧把它們縫合成一個世界。

## 項目三：選型軸反轉——這次「引函式庫」反而是低風險選項 (The Dependency Calculus Flips)

前篇（2D 天氣頁）結論是零依賴、拒用 WebGL；本篇結論相反：**採用 Three.js**。同一條原則的兩種結果——**真正的決策軸是「這條路徑在訓練分佈裡被踩得多熟」**，而非「依賴多寡」的教條：

- 2D 粒子：手寫 Canvas 是千錘百鍊的路徑，引函式庫反而增加變數 → 零依賴。
- 3D 場景：手寫原始 WebGL（矩陣、shader 管線、attribute 綁定）盲寫出錯面積巨大，且錯誤形態是無聲黑屏；Three.js 是被寫過最多次的 3D 函式庫，「用 Three.js 做 Minecraft」這條路徑本身有成千上萬篇教學 → 引依賴反而降險。

同時坦承 one-shot 高成功率的結構性原因：**不是在發明，而是在組裝典範解 (canonical solutions)**。Voxel 引擎、程序化地形（noise heightmap）、城市網格生成，都是有標準答案的已解問題。粗糙指令能成功，是因為指令命中了「解法密度極高」的題型；能力在於知道典範解是什麼、以及各典範解之間怎麼正確拼接。

## 項目四：效能是 3D one-shot 的生死線 (Performance as a Correctness Constraint)

2D 效能不好頂多掉幀；3D 方塊世界效能不處理是**直接不可用**。效能手段不是最佳化，是正確性的一部分，初版就必須內建：

1. **幾何合併 (geometry merging)**：絕不能一個方塊一個 mesh——一個區塊上萬 draw call 必死。整個 chunk 合併成單一 BufferGeometry，draw call 降到個位數。voxel 世界第一條軍規。
2. **隱藏面剔除**：只生成「暴露在空氣中」的方塊面，地底的面根本不進顯卡。通常削掉 90% 以上幾何量。
3. **視距與霧共謀**：只生成玩家周圍 N 個區塊，霧的濃度恰好讓 N 之外看不見——技術限制被氛圍系統偽裝成美術風格。
4. **GTA 場景同理**：建築用 instancing；車輛物理用簡化單車模型 (bicycle model) 而非真實剛體——「開起來像那回事」比「物理正確」重要且穩定。

**關鍵能力：** 對每個題型知道「不做就必死的軍規」是哪條。教學文章把這些放第八章，但 one-shot 沒有第八章，第一行就得按軍規寫。

## 項目五：互動閉環的完整性 (Closing the Interaction Loop)

「效果相當好」的另一半來自體感，體感來自互動閉環一次到位：

- **Pointer Lock ＋ WASD ＋重力跳躍**——第一人稱手感三件套，缺一件立刻露餡。
- **碰撞選型**：voxel 用 AABB 對格點碰撞——查表即可、數學上不可能穿模；拒絕通用物理引擎（多依賴、多盲寫參數）。選「不可能穿」的方案，不選「調好參數才不穿」的方案。
- **回饋即時性**：挖方塊有準星指示＋挖掉後地形立即重建。動作→世界改變的迴路閉合，「模擬器」的感覺才成立。

## 總結：此題型的三個特殊節點

1. **粗糙指令≠資訊不足**——遊戲名是高壓縮規格，勝負在「身份核心」砍得準不準。
2. **選型軸不是教條而是機率**——同一原則（最小化盲寫失敗率）在 2D 導出零依賴、在 3D 導出擁抱 Three.js。
3. **畫面品質是系統性的，不是逐物件的**——光照一致性＋霧＋色調統一把便宜的幾何縫合成「一個世界」；效能軍規（合併、剔除）讓這個世界跑得動。
