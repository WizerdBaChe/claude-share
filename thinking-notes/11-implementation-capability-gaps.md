# 思考鏈拆解：模型能力在實作上的系統性缺陷——三方方法論對讀後的缺口盤點

- **日期**：2026-07-12
- **任務設定**：將三份獨立產出的 one-shot 方法論（本系列 [01](01-oneshot-weather-web.md)、GPT 5.6 Sol 回覆、Gemini Flash 3.5 回覆）對讀，盤點「所有模型都沒講的」與「都講了但深度不足的」，作為後續 session 調整既有環境（規則層／坑卡層／模板層）的設計素材。
- **前篇對照**：[05](05-system-limits-and-biases.md) 談機器本身的規格瑕疵；本篇談**方法論文件層面的盲區**——即使模型把流程背得滾瓜爛熟，落到實作仍會系統性缺位的東西。
- **已定案前提**：對讀後已確認三個「可做成門檻的產出物」——驗收標準先於實作、選型否決表、不可驗證區標注。本篇不重複，只列其外的缺口。

---

## 甲部：明顯缺乏——三份文件都沒有的東西

### 缺口一：預算分配與降級順序 (Budget Allocation & Degradation Order)

三份文件都以「並列清單」呈現流程（GPT 十三章並列、Gemini 五階段並列），**沒有任何一份回答：當輸出預算不夠時，先砍哪一項？**

- 並列即無排序。實作中 token／注意力預算永遠有限，並列清單的實際行為是「每項各做六成」，而工程上正確的行為是「保命項做滿、裝飾項砍光」。
- 缺的不是優先級意識（01 有「失敗機率是第一排序鍵」），而是**明文的降級階梯**：預算吃緊時，砍霧效果之前先砍雷電？砍效果之前先砍動畫層？沒有一份文件把「不完整交付的形狀」設計出來。

**可操作化方向**：one-shot 類模板中增加一行強制產出——「降級順序聲明：若做不完，放棄順序為 X → Y → Z，核心保底為 W」。有這行就是做了，沒有就是沒做。

### 缺口二：規格猜錯的回收設計 (Spec-Miss Recovery Design)

三份文件都在需求還原階段下了重本，但**都預設自己的解讀是對的**。GPT 有「明示假設」一節（最接近），但只做到「宣告假設」，沒做到「讓錯誤假設便宜地可逆」。

- One-shot 沒有澄清回合，猜錯解讀是最大單點風險——而三份文件對此的全部防禦只有「猜得更準」。
- 缺的機制：**把解讀依賴的決策集中隔離**。例如「如果你要的其實是真實氣象資料而非展示效果，只需要換掉 `weatherSource` 這一個模組」——交付物自帶「如果我猜錯了，改哪裡」的地圖。
- 這與全域規則中的 swappable interface 條目同源，但那條的觸發是「品質存疑的元件」，此處的觸發是「解讀存疑的需求」——後者尚未被任何規則覆蓋。

**可操作化方向**：需求還原產出物中，每條「解讀」旁標注其在交付物中的**隔離點**（改哪個檔案／模組／參數可整體翻案）。

### 缺口三：失敗的可觀測性設計 (Failure Observability)

三份文件都在防「壞掉」，**沒有一份設計「壞掉的時候使用者會看到什麼」**。

- One-shot 交付後，唯一的除錯通道是使用者的回報。若失敗形態是無聲白屏，使用者只能回報「壞了」——第二輪等於盲修。
- 正確設計：讓每種失敗形態**自己會說話**——shader/資源載入失敗掛 error callback 顯示可見的錯誤條、console 有結構化錯誤訊息、關鍵初始化失敗時降級到靜態版本並標示。全域規則中的 GLSL shader-error callback 條目是這個原則的單一特例，原則本身未被提升。
- Gemini 的「邊界防禦」全是防止失敗發生，零字談失敗發生後的呈現；GPT 的驗證矩陣是作者自查，也不涉及此。

**可操作化方向**：交付前自問一題並寫進交付說明——「這個東西壞掉的三種最可能形態，使用者分別會看到什麼、該回報什麼」。

### 缺口四：第二輪經濟學 (Second-Round Economics)

01 的「預指最可能要調的旋鈕」是唯一觸及此的段落，但三份都**沒有把「最小化第二輪成本」當成明文的設計目標**。

- One-shot 的真實含義極少是「只有一輪」，而是「第一輪不能靠回饋」——第二輪（使用者開了頁面、提了修改）幾乎必然存在。
- 為第二輪優化的具體形態：可調參數集中在單一 config 區塊並附註解、美感參數（速度／密度／顏色）與結構邏輯分離、效果之間零耦合使單項重寫不動全局。這些散落在三份文件的架構章節裡，但動機都被歸給「工程整潔」，沒有被歸給「降低下一輪對話的成本」——動機不同，取捨就會不同（例如：為第二輪優化時，「參數命名的可溝通性」比「程式碼簡短」重要）。

**可操作化方向**：交付物尾端附「調整入口表」：想改什麼視覺感受 → 動哪個參數 → 合理範圍。

### 缺口五：對任務框架本身的抵抗 (Frame Resistance)

三份文件都**全盤接受了「ONE SHOT」這個框架**，沒有一份問：這個約束是真的嗎？

- 「一次做到完美」有時是使用者的真實約束（自動化管線、無人值守），有時只是修辭。若是後者，「先交一個問題清單再寫」的期望值可能遠高於「猜」。
- 這是 [05](05-system-limits-and-biases.md) 順從偏誤的方法論版本：**對任務前提的服從比對指令的服從更隱蔽**。三家在展示工程肌肉的同時，都沒展示「何時該推回框架」的判斷——而 [06](06-ask-vs-decide.md) 的四變數模型（可逆性／變異數／資訊位置／爆炸半徑）本可直接套用在「該不該接受 one-shot 約束」上。

**可操作化方向**：接到強約束（one-shot／急件／指定技術）時，第零步用一行判斷「此約束是實質的還是修辭的」，實質則遵守，存疑則以一句話確認——不展開成問卷。

### 缺口六：聲明—產物一致性檢核 (Claim-Artifact Consistency)

三份文件本身就示範了這個缺陷：**方法論聲明與實際可交付的機制之間存在裂縫，且模型自己不檢查**。

- 實例：Gemini 聲稱「靜態程式碼審查——逐行掃描擬定好的程式碼」（對尚未生成的程式碼做逐行審查，在自回歸生成下是修辭不是機制）；聲稱「無狀態分層模組」但架構明載一個狀態層（命名自相矛盾）；`will-change` 提示 GPU 加速對 canvas 元素效益邊際（cargo-cult 條目混在真條目裡，可信度被拉平）。
- 這是 [09](09-ai-reading-ai.md)「聲明 vs 證據」的自產自銷版：模型不只讀別人的輸出會被聲明騙，**寫自己的輸出時也會生成無對應機制的聲明**——因為「聽起來完整」在生成上比「確實完整」更順滑（[05](05-system-limits-and-biases.md) 自回歸約束的推論）。
- 三份都沒有內建「交付前把總結裡的每個方法論聲明對回產物裡的實體」這道檢核。

**可操作化方向**：交付總結中的每一條「我做了 X」必須可指向產物中的具體位置（檔案／函式／段落）；指不到的聲明刪除，不留修辭。

## 乙部：相通但深度不足——都講了、但停在背誦層

### 淺口一：心智模擬缺覆蓋率聲明 (Mental Simulation Without Coverage Claims)

三家都聲稱心智模擬（01「逐幀走查」、Gemini「沙盒模擬」、GPT 隱含在狀態模型章節），但**沒有一份劃出心智模擬的能力邊界**：

- 模擬**能**驗證的：狀態機轉移的完備性、生命週期配對（init/destroy）、邊界條件的分支覆蓋——離散邏輯。
- 模擬**不能**驗證的：實際幀率數字、視覺美感、浮點累積誤差、真實裝置的記憶體行為——連續量與感知量。
- 不劃界的後果：模擬過的討論全部獲得同等信心，而其中一半其實是未經驗證的猜測穿著「已模擬」的外衣。

**加強方向**：心智模擬的結論輸出時分兩欄——「邏輯上已窮舉」與「僅為經驗估計、需實測」。

### 淺口二：效能主張全數停在背誦 (Performance Claims as Recitation)

物件池、debounce、delta time、粒子數自適應——三家全講了，全對，**但全部止於「寫了這些技巧」，沒有一家讓交付物能證明技巧生效**。

- 缺的一步：交付物內建自我量測開關——按 `F` 顯示 FPS 與粒子數、切換天氣時 console 輸出清理前後的物件計數。成本約十行，收益是把「我做了效能優化」從聲明變成使用者可當場驗證的事實。
- 這同時服務缺口三（可觀測性）與缺口四（第二輪經濟學）：效能不好時，使用者回報的是數字而不是「感覺卡」。

**加強方向**：凡交付含動畫／粒子／即時渲染的產物，內建可開關的量測顯示，作為預設交付件而非加分項。

### 淺口三：邊界枚舉是靜態背誦而非環境查詢 (Static Boundary Lists vs. Environment Queries)

三家的邊界清單（resize／DPI／reduced-motion／背景分頁）高度一致——因為都是從訓練分佈背出來的**通用**清單。沒有一份先問：**這次的目標環境已知嗎？**

- 已知環境資訊（repo 裡的 browserslist、CLAUDE.md 記載的裝置、過往 session 提過「主要在手機上看」）應該覆寫通用清單的權重：明知只跑桌面 Chrome，還花預算寫觸控目標間距，就是把有限預算花在背誦上。
- 正確順序：先查已知環境事實 → 據此裁剪通用清單 → 只有環境完全未知時才全量套用。

**加強方向**：邊界枚舉前置一步「環境事實蒐集」（一次 grep／一眼 CLAUDE.md），清單按命中裁剪，並標注「因環境未知而保留的猜測項」。

### 淺口四：驗收清單寫給自己而非寫給非作者 (Author-Facing vs. Executor-Facing Checklists)

GPT 的驗證矩陣是三份中最完整的，但它是**作者自查表**（「功能：每個按鈕都切到正確天氣」）——一個沒讀過程式碼的人拿到它，不知道具體按什麼、預期看到什麼。

- 可執行驗收清單的規格：編號步驟＋每步的具體操作＋預期觀察結果（「3. 快速連點雨→雪→雨十次，預期：畫面上只有雪，無殘留雨滴，FPS 不低於初始值」）。全域規則的 manual-acceptance checklist 條目已有此要求，但三份文件顯示模型的**預設輸出形態**仍是自查表——規則在對抗預設，而預設會在規則未載入時回歸。
- 深一層的缺：清單沒有**否證性**設計。好的驗收步驟應該是「最可能翻車的操作」（連續快切、極端縮放、開著切分頁）而不是「正常使用路徑」——三家的清單都以正路徑為主，壓力路徑為輔。

**加強方向**：驗收清單以「非作者可盲執行」為格式標準，且壓力路徑條目數不低於正路徑。

### 淺口五：坑資料庫有存量無檢索紀律 (Pitfall Stock Without Retrieval Discipline)

三家都展示了坑存量（高刷屏 delta time、GC 卡頓、時間跳躍），01 更明言「坑資料庫前置命中」是勝負手。但**沒有一份談檢索紀律**：面對新任務，怎麼保證相關的坑被想起來？

- 坑的存量在訓練分佈裡大家都有；差異在**命中率**——而命中依賴觸發線索的組織方式。散文式的坑（藏在某篇筆記第五段）觸發率遠低於索引式的坑（「任務含 canvas 動畫 → 查 pitfall: canvas-*」）。
- 既有環境的 pitfall-card 機制正是解法，但目前卡片的觸發依賴描述行被讀到——缺一條「任務類型 → 必查卡片類別」的正向索引。

**加強方向**：坑卡層增加任務類型索引（canvas 動畫／跨 session 交接／subagent 委派 → 各自的必查卡集），把「希望想起來」變成「按類必查」。

## 落地優先序（供後續 session 調整環境時參考）

| 序 | 項目 | 落點層 | 理由 |
|---|---|---|---|
| 1 | 缺口六：聲明—產物一致性檢核 | 規則層（輕量） | 對抗的是生成機制的天然墮落方向，適用面最廣，一行規則即可表述 |
| 2 | 缺口一＋四：降級順序＋調整入口表 | one-shot 模板 | 產出物型門檻，可檢查；與已定案三產出物合併成同一模板 |
| 3 | 淺口五：坑卡任務類型索引 | 坑卡層 | 既有機制的增強，不佔規則預算 |
| 4 | 缺口三＋淺口二：失敗可觀測性＋內建量測 | 坑卡／模板層 | 具體技術實踐，適合做成卡片而非規則 |
| 5 | 淺口三：環境查詢先於邊界背誦 | 模板層 | 一行前置步驟 |
| 6 | 缺口五：框架抵抗 | 不新增 | [06](06-ask-vs-decide.md) 判斷框架已覆蓋，僅需在該篇補一句「約束也是待驗前提」 |
| 7 | 淺口一＋四 | 併入既有規則 | manual-acceptance 與 R8 已有掛點，補充措辭即可，不開新條 |

**總原則（重申且適用於本篇自身）**：只編碼模型預設不會做的事；凡無法對應到可檢查產出物的項目，寧可留在本篇當參考，不升格為規則——系統瓶頸是規則預算，不是流程存量。
